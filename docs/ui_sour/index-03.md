# UI SOUR

UI SOUR（ユー・アイ・サワー）は、特定のレシピや機材構成、固定された手順を指す名称ではありません。  
**味（カクテル）・音（演奏）・身体（生理的変化）**が相互に規定し合うフィードバックのなかで、必然的に生じるノイズを扱うための**オープンなフレームワーク**です。

- **フレームワーク**：実装（具体的な素材・機材・手順）を固定せず、複数の実装が成立するための枠組み。
- **ノイズ**：外部から混入する異物ではなく、関係の運動そのものが生む「ずれ」。
- **相互規定**：AがBを一方的に決めるのではなく、AとBが互いを条件づけ合うこと。
- **フィードバック**：結果が次の条件として戻り、更新が継続する循環。

---

## 指示書（最小公理）

UI SOURの作品指示書は、次の一文のみです。  
詳細は `framework.md` に固定し、ここでは参照として示します。

> 味・音・身体の相互規定的フィードバックが必然的に生成するノイズを引き受けよ。  
> それらを、また更新し続けよ。

- **更新**：判断と介入の痕跡を保持したまま、次の循環の条件を組み替えること。

→ **Framework（指示書本文）**：[`framework.md`](framework.md)

---

## このリポジトリに置くもの

UI SOUR は「単一の正解」を持ちません。代わりに、次の二層で記述を管理します。

1. **Framework（フレームワーク）**  
   最小公理（指示書）と、フレームワークとしての境界条件を記述します。  
   → [`framework.md`](framework.md)

2. **Interpretations（解釈／実装例）**  
   具体的な素材・機材・手順・記録形式は、各実践者による**解釈**として保存します。  
   - **解釈**：フレームワークを満たす範囲で、具体の手順や対応関係を一つに定めること。  
   - **実装**：解釈を実際に運用できる形（文章・表・ルール）に落とし込んだもの。

## 使い方（最短）

1. まず `framework.md` を読む（最小公理と境界条件）。  
2. `interpretations/` から任意の解釈を選ぶ、または自分の解釈を作る。  
3. 実施後は、判断と結果を次の循環へ持ち越し、必要なら解釈（手順）自体も更新する。  
   その更新は、UI SOUR の外ではなく、UI SOUR の内側で起こるものとして扱います。
